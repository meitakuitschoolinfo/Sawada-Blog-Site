# セキュリティ／非機能設計メモ

| システム名 | Blog アプリケーション |
| --- | --- |

本書は、現状のソースコードから読み取れるセキュリティ・非機能面の**実態（As-Is）**と、**改善案（To-Be）**を整理したものです。
本アプリは学習用のため、実運用で必須となる対策の一部が未実装です。

---

## 1. 認証・認可

| 項目 | As-Is（現状） | リスク | To-Be（改善案） |
| --- | --- | --- | --- |
| ログイン認証 | `email` + `password` の完全一致で判定（`findByEmailAndPassword`） | パスワード平文照合 | Spring Security の導入、ハッシュ照合 |
| 認可（アクセス制御） | `/user/blog/**` は未ログインでも URL 直打ちでアクセス可能。各画面でセッションの `user` を前提に参照 | 未ログイン時に `NullPointerException`、情報露出の可能性 | 認証フィルタ／インターセプタ、または Spring Security でのルート保護 |
| 認可の所有者チェック | 編集・削除時に「その記事が本当にログインユーザーのものか」を検証していない（`blogId` のみで操作） | 他ユーザーの記事を ID 指定で操作できる恐れ | `userId` と記事の所有者を突き合わせて検証 |

---

## 2. パスワード管理

| 項目 | As-Is | リスク | To-Be |
| --- | --- | --- | --- |
| 保存形式 | **平文**（`users.password` にそのまま保存） | DB 流出時に即漏洩 | ソルト付きハッシュ（BCrypt 等）で保存 |
| 参考実装 | `UserService` 内に SHA-256 + ソルトのサンプルがコメントで残存 | - | `BCryptPasswordEncoder` の利用を推奨 |

---

## 3. 入力・ファイルの取り扱い

| 項目 | As-Is | リスク | To-Be |
| --- | --- | --- | --- |
| 入力バリデーション | サーバー側の項目チェックは未実装（`@RequestParam` バインドと業務チェックのみ） | 不正値・空値の混入 | Bean Validation（`@Valid` / `@NotBlank` 等）の導入。詳細は [17_バリデーション仕様書.md](17_バリデーション仕様書.md) |
| 画像アップロード | 拡張子・MIME・サイズの検証なしで `static/blog-img/` に保存。ファイル名は日時 + 元ファイル名 | 不正ファイル混入、容量枯渇、同名衝突の可能性 | 拡張子/サイズ検証、保存先の外部化、一意キー名の採用 |
| 保存先 | アプリの `src/main/resources/static/` 配下に直接保存 | ビルド成果物と混在、再デプロイで消える | 外部ストレージ／専用ディレクトリへ分離 |

---

## 4. セッション管理

| 項目 | As-Is | To-Be |
| --- | --- | --- |
| ログイン状態 | `HttpSession` に `UserEntity` を `"user"` キーで保持 | 必要情報のみ（userId 等）を保持 |
| ログアウト | `session.invalidate()` で破棄 | 妥当（現状で問題なし） |
| CSRF 対策 | 未対応（POST フォームに CSRF トークンなし） | Spring Security 導入で CSRF トークンを有効化 |

---

## 5. その他非機能

| 分類 | As-Is | To-Be |
| --- | --- | --- |
| 例外処理 | DB 例外は基本的に上位へ伝播。ファイル保存は try-catch でスタックトレース出力のみ | 共通例外ハンドラ（`@ControllerAdvice`）とエラー画面 |
| ログ | `spring.jpa.show-sql=true` / Hibernate SQL を DEBUG 出力 | 本番では SQL ログを抑制、アクセス/監査ログを整備 |
| 設定情報 | DB パスワード等が `application.properties` に平文記載 | 環境変数／Vault 等で機密分離、プロファイル分割 |
| トランザクション | 削除のみ `@Transactional`。更新系はメソッド単位の暗黙トランザクション | サービス層でのトランザクション境界を明示 |

---

## 6. 優先対応の提案（学習ステップ）

1. **パスワードのハッシュ化**（`BCryptPasswordEncoder`）
2. **認可ガードの追加**（未ログイン時のリダイレクト、記事所有者チェック）
3. **入力バリデーションの導入**（Bean Validation）
4. **Spring Security の導入**（認証・認可・CSRF をまとめて担保）

> 本書は「現状の指摘」と「改善方向」を示すものであり、現行コードの動作を変更するものではありません。
